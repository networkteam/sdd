The llm.Runner currently returns only the response text, discarding all metadata from claude -p. Switching to --output-format json exposes per-call token counts (input, output, cache read/creation), cost (total_cost_usd), and timing (duration_ms, duration_api_ms) — everything needed to diagnose why pre-flight is slow and where tokens go. The runner should parse the JSON response, extract the result text for callers, and log the metadata via log/slog structured logging. In normal mode: silent. In verbose/debug mode: per-call structured log lines with entry ID, token counts, cost, and duration. This gives observability into both pre-flight and summary generation without changing the generation interface — callers still get text back, but operators can see what the LLM layer is actually doing.
