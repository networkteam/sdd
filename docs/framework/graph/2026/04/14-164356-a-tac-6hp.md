Implemented the entry summary infrastructure decided in d-tac-h93. New llm package (sdd/llm) owns all LLM generation — preflight moved from finders for consistent architecture, summary generation added alongside it. Each entry gets a summary frontmatter field (2-3 sentences describing entry and direct relationships) and summary_hash (SHA-256 of the full rendered prompt — template, content, and upstream summaries all contribute, so any change automatically invalidates). The sdd summarize command processes entries in DAG topological order, skips entries whose hash matches, and commits in one batch. Lint detects missing or stale summary hashes. Summaries are also generated automatically at sdd new time after preflight passes. Dialogue refinements beyond the original plan: summary_hash was pulled forward from deferred to first implementation; hash is computed from the rendered prompt rather than separate input fields; preflight was fully moved to llm package (user decision — half-way refactoring is always bad). Deviation (user-approved scope split): CLI output changes (list/show/status consuming summaries) and skill updates (catchup/explore) are handled by the parallel session implementing depth-limited BFS traversal (d-tac-017). This is an intentional division of scope, not a reduction — d-tac-017 depends on summaries being populated (which this action delivers) and owns the consuming side (how summaries are rendered in commands and fed to sub-skills). The user decided this split during dialogue because the recursive max-depth changes in d-tac-017 fundamentally reshape how commands and skills consume entry data, making it the natural home for all consumption-side changes.
