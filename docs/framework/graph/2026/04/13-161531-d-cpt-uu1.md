Distribute SDD via GoReleaser → GitHub Releases → custom Homebrew tap (e.g. networkteam/homebrew-sdd) for brew install UX, complemented by a binstaller-generated install.sh (config-driven from the GoReleaser spec, embedded checksums, optional gh attestation verify chain-of-trust via actionutils/trusted-go-releaser) for non-brew users. Both paths are agent-neutral per the agent-platform-neutrality directive d-stg-574. Update story: Homebrew users run brew upgrade sdd; curl|bash users re-run the same idempotent installer command. A self-update subcommand (sdd self upgrade, analogous to rustup self update / uv self update) is deferred post-MVP — the two channels both have working update paths without it. Skills ship embedded in the Go binary (via go:embed) and are extracted by sdd install into the target agent's skills directory (default ~/.claude/skills/ at launch; --scope project and --target <agent> flags designed in from day one even if only Claude is wired up at MVP — this is the foundation/feature-surface split d-stg-574 specifies). Versioning has three coordinated stamps: ldflags-stamped binary version, per-skill sdd-version frontmatter stamp set by sdd install, and a graph metadata file (docs/framework/graph/.sdd-meta.json holding graph_schema_version and last_touched_with). Schema compatibility uses Option A+D: the main binary is strict on graph_schema_version with a clear error pointing at sdd migrate; migrations are bundled in the binary and chain transitively (v1→v2→v3) so users never need a historical sdd version to upgrade across multiple schemas. No macOS notarization at launch — Homebrew handles Gatekeeper for brew users; curl-installed binaries to ~/.local/bin/ don't trigger Gatekeeper either (no com.apple.quarantine xattr is set by curl, session-verified against binstaller's own self-installer on macOS). Claude Code plugin marketplace is deferred as an optional secondary channel (discovery-only or with a SessionStart hook to fetch the binary), never the primary path — this flows from d-stg-574. Sequencing: fork to dedicated SDD repo first per d-cpt-dlw, then set up distribution infra in the new repo. Research chain: findings from the seven-angle landscape scan (see research action a-cpt-xbv) drove the Go-CLI-ecosystem convergence on GoReleaser + Homebrew tap + curl-style installer pattern, the two-stage install as the idiomatic bootstrap, and bundled skills as the robust answer to version skew. Session-introduced binstaller substitutes for hand-written install.sh and adds attestation chain-of-trust as a security upgrade. Alternatives considered and rejected (full set in plan.md): Claude Code plugin marketplace as primary (violates d-stg-574); embed-only with no conventional distribution (acquisition story unsolved); Apple Developer enrollment at launch (unjustified cost); multi-version graph reader/writer (Option C, complexity compounds); Homebrew core formula at launch (longer review with no MVP UX gain). See [plan](./13-161531-d-cpt-uu1/plan.md) for full implementation shape, install UX with all three acquisition variants, versioning mechanics, schema compat details, update story per channel, alternatives rejected with reasoning, deferred sub-questions, and rough implementation ordering.
