Landed confidence-scored pre-flight verdicts and inline plan ACs per d-prc-g1h across four phases, with three notable deviations driven by mid-implementation dialogue and eval runs: (1) output format switched from bullet lines to JSON `{"findings": [...]}` after eval runs showed the bullet format caused LLM inconsistency (code fences, commentary, sentinel ambiguity on identical inputs); (2) acceptance criteria moved inline into plan decision descriptions rather than attachments, dropping the `AcceptanceCriteria` / `ProposedAttachments` plumbing entirely — ACs flow through `ClosedEntries` / `ProposedEntry` automatically via `FormatEntryForPrompt`, driven by the observation that embedding full plan attachment content in the prompt was token-heavy and the extraction plumbing was brittle; (3) AC coverage calibration clarified to "reasoning presence, not quality" after the silent-vs-ambiguous split left a gap where "explicitly not done without why" fell through and the LLM flipped severities — refinement stabilized judgment across the silent/named/deviated scenarios in eval. Full AC coverage detail in the attachment ([coverage](./15-152324-a-prc-84r/coverage.md)). Code in commits bf87f16, 55ef589, d1891c7, 4c09be6, 3e7b98d, ecd367c. Eight-test eval suite passes consistently across severities and check types. Deferring AC 15–16 (full corpus-based suite of 13 FP + 16 TP cases) is a dialogue decision: the smoke suite validates core behavior, pre-flight runs every `sdd new` so production usage will surface new signals naturally, and the full run is ~15–20min + real API cost — expanding lazily based on observed failures beats speculative coverage.
