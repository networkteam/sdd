sdd new and pre-flight disagree on default field handling. When sdd new is invoked without an explicit --kind flag for a decision, the kind defaults to directive semantically but is not written to the entry's frontmatter. The pre-flight validator reads the raw frontmatter without applying defaults, so it sees the kind field as missing. If the entry description explicitly mentions its kind (e.g. "starting as a directive, not kind: contract" in d-stg-574), pre-flight rejects with: "Missing Kind field in header (decision text specifies X but Kind is not included)." Workaround applied during d-stg-574 capture: explicitly pass --kind directive even though it is the default, which writes the field to the frontmatter and satisfies pre-flight. The same description text was rejected without --kind and accepted with --kind directive, isolating the cause to the file representation rather than the content. Two possible fixes: (a) the CLI always writes kind: directive to the frontmatter so the file is self-describing and matches what pre-flight sees; or (b) the pre-flight validator computes defaults before validating, matching how readers interpret the file. Either resolves the writer/validator view inconsistency. Related observation captured separately from this signal: the --preflight-timeout flag is typed as int (seconds) where time.Duration would be friendlier (urfave/cli has a DurationFlag). Both are small operational gaps in the CLI surface that the in-flight CQRS migration (d-tac-lwm) is well-positioned to address — refactoring the new-entry handler is a natural moment to make defaults explicit at the writer side, and the flag-type fix is independent and trivial.
