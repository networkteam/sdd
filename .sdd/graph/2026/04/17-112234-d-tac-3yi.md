Render each entry's computed state as a derived attribute alongside stored attributes, so the proposals-vs-facts distinction established in framework-concepts is visible at a glance.

Observation: agents confuse open signals as already implemented — cited a signal's proposal as live behavior this session (captured as s-prc-sda, closed by a-prc-fxn on the concept side).

Hypothesis: agents are the primary audience affected by this gap. Humans navigate the graph with richer context and more skepticism; agents skim mechanical attributes and miss state that isn't rendered there.

Scope: compute each entry's state from downstream relationships and render it inline wherever an entry ID appears. States: `{status: active}` (decision / plan / contract, not superseded or closed), `{status: open}` (signal, not superseded or closed), `{status: closed-by <id>}`, `{status: superseded-by <id>}`. Surfaces: `sdd status`, `sdd list`, `sdd show` headers, upstream/downstream chains, grooming tables, lint output.

Design choices:
- Distinct bracket notation for stored vs derived attributes. Stored frontmatter fields use square brackets (`[kind: plan] [confidence: medium]`); derived attributes (computed from graph traversal) use curly braces (`{status: active}`, `{status: closed-by a-prc-fxn}`). The visual split tells a reader at a glance which attributes are written on the entry and which come from its relationships.
- Always keyed derived attributes. `{status: active}` over terser `{active}` because the `status:` key is mechanically scannable — agents don't need a dictionary of bare labels, and future derived dimensions (semantic kinds from d-cpt-omm's two-type restructure, graph-traversal categorizations) extend by adding new keys (`{kind: gap}`, `{thread: ...}`) without collision.
- Include the closing/superseding ID in the value (`{status: closed-by a-prc-fxn}`) so readers follow to "what was done" in one step, without traversing Downstream manually. Always full IDs for determinism. Space separator inside the value (not colon) because the outer `{key: value}` colon would collide.
- `sdd status` renders the derived attribute even though its membership already implies state — consistency matters more than the redundancy; agents shouldn't need different reading rules per surface.
- Status is derived, not a frontmatter field. Entries stay immutable; state is recomputed on every read. The framework reference should note this so no one edits entries to "update" state.

Positive complement to s-prc-sbl's fix (synthesis out of sub-skills): surface state as data, don't ask the agent to derive it.

## Acceptance criteria

- [ ] Stored frontmatter attributes render with square brackets (`[kind: plan] [confidence: medium]`) across `sdd status`, `sdd list`, `sdd show` headers, and summary-line chains
- [ ] Derived state renders as `{status: active}`, `{status: open}`, `{status: closed-by <full-id>}`, or `{status: superseded-by <full-id>}` in the same surfaces
- [ ] `sdd status` renders the `{status: ...}` attribute on each entry
- [ ] `sdd list` renders `{status: ...}` on every entry, including closed / superseded under `--all`
- [ ] `sdd show` includes `{status: ...}` in the header block for the primary entry
- [ ] Summary-line format for upstream / downstream chains carries `{status: ...}` next to each entry ID
- [ ] Grooming-candidate tables from `/sdd-groom` surface `{status: ...}` in the Status column (or equivalent)
- [ ] The curly-brace value for compound states uses a space separator (`closed-by <id>`, `superseded-by <id>`), not a nested colon, to avoid ambiguity with the outer `{key: value}` delimiter
- [ ] `references/framework-concepts.md` notes that derived attributes (curly-brace notation) are computed from graph relationships, not stored on the entry — entries remain immutable
