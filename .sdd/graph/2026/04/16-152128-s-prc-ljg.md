Pre-flight severity should be phase-aware: the same contract produces different severity findings depending on whether it is evaluated during plan capture (medium — advisory), implementation start via wip start (high — blocking), or closing action (high — blocking). This lets project-specific contracts act as escalating guardrails without SDD imposing a fixed process — the graph's own decisions drive the process gates.

The problem this addresses: tactical plans naturally stay at the what/why level, but projects may have contracts (like d-cpt-ah1 requiring CQRS decomposition) that demand detailed architectural thinking before implementation. Currently there is no gated step between plan and implementation — an agent can go straight from a high-level plan to writing code, bypassing decomposition that contracts require. Catching this only at closing time means the wrong thing was already built.

Pre-flight on wip start fills this gap. When an agent begins implementing a plan, pre-flight evaluates active contracts against the plan's current state. If a contract expects decomposition (operational sub-decisions, architectural breakdown) and none exists, it blocks: Contract d-cpt-ah1 expects CQRS decomposition before implementation — capture operational decisions first. The agent must stop and dialogue about the missing layer before building.

For this to work, the SDD skill must guide agents to call wip start before implementing — making it the standard transition point from planning to action. The skill already recommends this but it needs to be strengthened as the expected path. If dialogue determines wip start is unnecessary for a specific case (small change, obvious path), the agent can skip it — but the closing action pre-flight still evaluates contracts at high severity as a safety net, so contract violations are caught regardless.

This is project-driven, not SDD-imposed. A project without architectural contracts gets no implementation gate. A project with heavy contracts gets proportionally more pre-flight scrutiny at implementation start. Projects shape their own process through their own graph entries — SDD just provides the phase-aware evaluation mechanism.
