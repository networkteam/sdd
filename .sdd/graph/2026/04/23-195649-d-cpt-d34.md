Refines the actor and role type-system implementation plan (supersedes `d-cpt-thg`) with three material additions — a role-status cascade from the actor-identity chain's canonical history, a `sdd status` Participants block for native role awareness in catch-up, and a `sdd lint` orphan-role check — plus a refinement to the existing role-entry pre-flight rubric that specifies the **current head** canonical, making the capture-time vs derivation-time split explicit. Acceptance criteria grow from 15 to 18 entries; the framework-concepts and unit-test ACs are restated more granularly to name the new surfaces (cascade scenarios, Participants-block rendering). The CQRS decomposition, retirement path, grace-period semantics, and LLM-validator retirement are preserved; refinements concentrate on role lifecycle and presentation surfaces. All ACs from `d-cpt-thg` carry over (refined or restated where noted) — no AC is silently dropped. See [dialogue path and alternatives](./23-195649-d-cpt-d34/plan.md) for the reasoning behind the changes.

Extends the SDD type system with dedicated `actor` signal and `role` decision kinds (per `d-cpt-pc0`), landing them across model, handlers, pre-flight, lint, status rendering, and skill documentation. AC 1 directs the creation of a new contract decision that supersedes the current two-type kind contract (`d-cpt-ydf`) with a 6+6 kind system — the contract supersession is enacted by that new entry at implementation time, not by this plan. Wires mechanical canonical validation for participants (`d-cpt-979`) and role `actor:` refs, with a self-transitioning grace period so fresh graphs aren't blocked before any actors exist. Adds `sdd lint` participant-coverage, alias-ambiguity, invariant, and orphan-role checks that surface unresolved names and graph integrity violations — the signals the downstream bootstrap playbook plan (`d-cpt-kv1`, a consumer that refs this plan) relies on to dialogue each gap into an actor capture.

CQRS decomposition (per `d-cpt-ah1`): `model` extends kind enums and adds actor/role domain types, the chain-canonical-history index, and — new vs `d-cpt-thg` — the role-status derivation function (cascade logic computing active/closed from a chain's canonical history). `finders` gain a participant-coverage finder, an active-actor finder, and a role-status finder; all three orchestrate entry loading and delegate the pure computation to model functions (per the push-logic-down rule in `CLAUDE.md`). `handlers` extend participant validation in the new-entry handler with the mechanical canonical check; pre-flight templates gain kind-specific rubrics for actor and role; `presenters` extend list/status/show rendering for the new kinds plus the new Participants block in `sdd status`; no new CQRS commands (creation reuses existing `new` with `--kind` flag).

**The role-status cascade logic lives in the model layer**, following the push-logic-down rule (`CLAUDE.md`). The chain-canonical-history index and the derivation function that computes active/closed from it are both pure model concerns — no side effects, no I/O. The role-status finder is a thin orchestration wrapper that loads entries and delegates to the model function, mirroring how the active-actor finder delegates chain traversal to model primitives.

Historical entries stay untouched (`d-cpt-e1i`) — existing kinds remain valid, and pre-flight's canonical check applies only to new captures. Lint surfaces historical participant gaps for the bootstrap playbook to resolve dialogically rather than forcing retroactive modifications.

This plan refines the two-layer commitment in `d-cpt-979` (LLM-judged semantic check + mechanical structural check for participants) to a single-layer purely mechanical check, following further dialogue that recognized canonical-name uniqueness plus a capture-time canonical-only discipline makes the LLM layer unnecessary. Agents and skills always write canonical forms in `participants`; alias resolution is a read-side concern for mining, dialogue comprehension, and external-source import — never a validation-time concern. The existing LLM-judged participant validator (`s-tac-7ab` groundwork, originally from plan `d-tac-6z1`) is retired by this plan's implementation in favor of the mechanical canonical check.

Canonicals are write-once across actor-identity chains: a canonical can only appear in one supersede chain ever. Within a chain, canonicals are free (typically shared across supersessions, occasionally changing for corrections). Across chains — including after a chain is closed — canonicals are retired and cannot be reused. This preserves temporal stability of historical entries' participant references without requiring per-entry timestamp-based name resolution. (Same semantics as `d-cpt-thg`; no change.)

**Role status cascades from the actor-identity chain's canonical history.** A role with `actor: X` resolves to the chain that has ever held X as canonical (unique by the write-once-across-chains invariant). The role is active iff that chain's head actor signal is active. Retiring an actor chain transitively closes all roles binding to any canonical in the chain's history — no separate role-retirement entries needed. This keeps the role's `actor:` value stable under within-chain canonical corrections, since derivation walks full chain history rather than matching current head only. Capture-time validation uses the current head's canonical to force new captures to the latest form; derivation-time resolution uses full chain history to keep old captures valid. Capture-time and derivation-time are complementary, not redundant — both layers remain in force.

**Role awareness enters agent context natively via the `sdd status` Participants block.** A new block after existing status sections groups by active-actor canonical appearing in shown entries, with active role entries listed underneath in the standard entry-line format used by other status sections (consistent with agent-first CLI design). The block is suppressed during grace when zero active actors exist. This scopes role context to the active-participant set rather than listing the full role inventory.

The orphan-role lint check handles abnormal state distinct from the normal-case cascade: a role entry whose `actor:` canonical doesn't match any chain's history (e.g., a direct file edit that bypasses pre-flight, a corrupted entry, or a validator bypass). The cascade handles normal retirement automatically; the orphan check surfaces pathological entries that shouldn't exist in a healthy graph.

Grooming treats actors and roles as ordinary entries — the existing flow handles stale-actor and stale-role candidates via its standard patterns. No special handling required.

Follow-up carved out as future work: summary of active-participant roles beyond what the Participants block surfaces (e.g., role-scoped catch-up clustering, decision-time role hints). Deferred pending felt need after adoption.

## Acceptance criteria

- [ ] `d-cpt-ydf` superseded **per `d-cpt-pc0`** via a new contract decision created as part of this plan's implementation: the new contract expresses 6 signal kinds (`gap`, `fact`, `question`, `insight`, `done`, `actor`) and 6 decision kinds (`directive`, `activity`, `plan`, `contract`, `aspiration`, `role`)
- [ ] Actor signal kind implemented **per `d-cpt-cye`**: `canonical` required + `aliases` optional in frontmatter, process layer, lifecycle (supersede on identity-fact change / close via retirement directive), default confidence high, external context in prose narrative. `sdd new s prc --kind actor` creates valid entries.
- [ ] Role decision kind implemented **per `d-cpt-9dm`**: `actor` (canonical name) required in frontmatter, process layer, lifecycle (supersede on participation-pattern shift / close via retirement directive), default confidence medium, multiple-roles-per-actor permitted, prose scope/content (no structured action taxonomy). `sdd new d prc --kind role` creates valid entries.
- [ ] Pre-flight templates include kind-specific rubrics for actor and role **per `d-cpt-cye` / `d-cpt-9dm`**: actor rubric expects canonical + optional external context prose; role rubric expects actor canonical + current-actor-entry-id in refs + contribution-shape prose.
- [ ] Actor canonical write-once invariant: pre-flight on actor captures rejects a new actor whose `canonical` has ever been used as the canonical of an actor signal in any other actor-identity chain. Within the supersede chain the new entry extends, canonicals can be shared or changed across entries. Closed actor chains do not free their canonicals for reuse.
- [ ] Pre-flight mechanical canonical check on new non-actor entries **per `d-cpt-979`**: each name in `participants` must match the canonical of exactly one active actor signal (no alias resolution in the validation path). Binary severity (blocks on miss). Self-transitioning grace: skipped when graph has zero active actor signals, activates once the first actor is captured.
- [ ] Pre-flight mechanical check for role entries **per `d-cpt-9dm`**, retaining both checks from `d-cpt-thg`'s original AC (the capture-time rubric): (a) the role's `actor:` canonical matches the **current head** canonical of an active actor-identity chain, AND (b) the role's `refs` list includes the entry ID of that chain's head actor signal. The refinement vs `d-cpt-thg` is the addition of the word "current head" on (a) — the canonical must match the chain's active head, not a historical canonical; requirement (b) is unchanged. Both checks are capture-time; derivation-time behavior is governed by the separate role-status cascade AC below.
- [ ] **Role status derivation** (new AC — derivation-time behavior, complementary to AC 7's capture-time checks): a role with `actor: X` resolves to the actor-identity chain that has ever held X as canonical (unique by write-once-across-chains). Role is derived-active iff the resolved chain's head actor signal is active; retiring the actor chain transitively derives-closed all roles binding to any canonical in the chain's history. No separate role-retirement entries required for the cascade. Derivation logic lives in the model layer (per push-logic-down); the role-status finder delegates to model functions.
- [ ] LLM-judged participant validator retired: pre-flight template, finder logic, and handler wiring originally delivered by `d-tac-6z1` / `s-tac-7ab` are removed in favor of the mechanical canonical check. Historical drift captured in the plan description.
- [ ] `sdd lint` participant-coverage check: surfaces every distinct participant name in the graph that does not match an active actor canonical, with the list of entries referencing each name. Drives bootstrap-playbook dialogue resolution.
- [ ] `sdd lint` alias-ambiguity check (informational, non-blocking): surfaces aliases colliding across active actor signals so the skill can flag them when encountered at read/mining time.
- [ ] `sdd lint` invariant check: detects any canonical appearing across different actor-identity chains (violates write-once). Defense-in-depth against race conditions or validator bypass.
- [ ] **`sdd lint` orphan-role check** (new AC): flags roles whose `actor:` canonical does not appear in any actor-identity chain's canonical history. Handles abnormal state (direct file edits bypassing pre-flight, corrupted entries, validator bypass) — distinct from the normal-case cascade where roles derive-close automatically when the actor chain retires.
- [ ] `sdd list --kind actor` and `sdd list --kind role` filter correctly.
- [ ] **`sdd status` Participants block** (new AC): after existing sections, renders each active-actor canonical appearing in shown entries as a group header, with active role entries listed underneath in the standard entry-line format used by other status sections. Block is omitted during grace when zero active actors exist. Role cascade derivation governs which roles appear — retired actor chains hide their roles automatically.
- [ ] Framework concepts reference (`framework-concepts.md`) documents actor/role kinds and semantics **per `d-cpt-pc0` / `d-cpt-cye` / `d-cpt-9dm`**, including: orthogonality of role to contract, canonical-only rule for `participants` at capture time, alias-as-read-side-convenience, canonical write-once invariant, and the role-status chain-canonical-history cascade. (Scope vs `d-cpt-thg`: extended to name the cascade semantics explicitly.)
- [ ] Main `/sdd` `SKILL.md` gains guidance **per `d-cpt-979`**: write canonical names in `participants` (never aliases), aliases used only for dialogue comprehension and mining, do not auto-ref actors/roles for routine participation. Notes that actor/role entries surface via the new `sdd status` Participants block (and `sdd list --kind actor|role` for filtered views), not the existing kind-based status sections.
- [ ] Unit test coverage carries over all scenarios from `d-cpt-thg`'s unit test AC (actor/role creation, supersede, close via retirement directive, canonical write-once invariant check, pre-flight mechanical canonical check for new entries + role entries, lint participant-coverage + alias-ambiguity + invariant checks) **plus new scenarios for this plan's additions**: role-status chain-canonical-history cascade (typo-correction-within-chain, chain-retirement), `sdd lint` orphan-role check, and `sdd status` Participants block rendering (grace suppression, role cascade, entry-line format consistency).
