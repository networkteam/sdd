Bidirectional sync is incomplete: background fetch and conflict prediction are shipped, but push remains a manual user step (the `sync: local ahead by N, consider push` suggestion), so graph changes stay unpublished by default. This gap asks whether the `sdd` CLI should somehow help with push after commit-producing operations (`sdd new`, `sdd init`, `sdd summarize`, `sdd wip start/done`), and how to handle the boundary where the local branch carries non-sdd commits that aren't ready to publish. One initial sketch worth thinking through: a config-level auto-push mode with options like `never`, `always`, or `safe` (push only when all unpushed commits are sdd-prefixed) — the existing `^sdd:` commit pattern could plausibly support the safe variant, but the trade-offs (silent network calls, partial pushes, mixed-branch states, what "safe" really means) need dialogue before any direction is chosen. Refines the background sync plan (d-tac-hsu) and the shipped sync awareness (s-tac-2y0) by surfacing the publish side of the bidirectional flow as an open question.
