Refs currently carry only an entry ID and relationship type (refs / closes / supersedes), with the reason each ref exists living only in the entry's narrative text. When presenting d-prc-g1h's refs during capture dialogue, annotating each with a one-line 'why' (e.g. 'a-prc-dix — empirical grounding for calibration') significantly improved scannability. Proposes optional per-ref annotations captured as structured metadata — format to be decided (inline parenthetical vs. frontmatter object). Value: catch-up and exploration render the 'why' per relationship, reducing the need to chase upstream chains to understand motivation; pre-flight context assembly can include the rationale alongside the referenced entry's summary; narrative prose can lighten since ref rationales aren't duplicated. Trade-offs: frontmatter format complexity, parser updates across sdd show / sdd list / sdd-catchup / sdd-explore / lint, immutability friction (annotations frozen at capture time can go stale if the nature of the relationship evolves), and optional-vs-required tension. Worth dialogue on format and scope before implementation.
