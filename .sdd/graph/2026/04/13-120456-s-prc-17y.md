Pre-flight caught a real implementation bug during the closing action for d-tac-q5p — dry-run + early validation failure didn't save stdin, contrary to the plan's "save on every dry-run (pass or fail)." The bug was fixed in a follow-up commit, but no regression test accompanied the fix. The closing action's own description rationalised this with "exercised manually because the Action closure is tightly bound to urfave/cli plumbing" — which functioned as a permission slip for skipping the test. User input during retrospective: it doesn't matter how the bug was reported (pre-flight, manual, production, post-merge review) — every fix needs a test that captures the regression. Test-first is advised when practical (write failing test, implement fix, watch it pass) but isn't strictly required; the discipline is "every fix has a test," not "every fix is TDD." A secondary benefit: forcing yourself to reproduce a regression in a test pushes the code toward better testability — "hard to test" stops being a free pass and starts being a signal that the structure should change.
