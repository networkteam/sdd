SDD distribution tooling is unsolved for external use. The Go CLI at framework/bin/sdd works in-repo, but there is no acquisition path for external users: no binary release flow, no version story, no update mechanism. The skill suite (sdd, sdd-catchup, sdd-groom, sdd-explore) also lacks a delivery mechanism — skills currently live in the repo's .claude/skills/, not portable. Without solving this, s-stg-qg0 (adoption paths) can't progress past 'works on my machine' and the fork decision (d-cpt-dlw) lands an empty-shelf repo for external evaluators. Three options are on the table with different trade-offs: packaging as a Claude Code plugin (native fit, unknown binary-handling story), embedding skills inside the sdd binary and using sdd to install them into projects (single artifact, unclear Claude Code skill-discovery compatibility), or a curl|bash installer (simplest, no update path). All three need research before a decision — binary release flow, macOS code signing, and platform builds are cross-cutting concerns regardless of which path wins. See [options breakdown](./13-143855-s-cpt-4gj/options.md) for details.
