Plan for stdin persistence and `--dry-run` flag on `sdd new`, so users can iterate on entry framing without re-piping heredoc content between pre-flight attempts. Supersedes d-tac-m94 with revised scope (moves `.sdd-tmp/` under graph-dir, drops auto-cleanup) and adds the `--dry-run` flag from s-tac-7ze. Implementing together because both share the stdin-save logic and address the same iteration friction.

**Stdin persistence on rejection:**
- When `sdd new` receives an attachment via `-:filename.md` and pre-flight rejects, write stdin bytes to `<graph-dir>/.sdd-tmp/<hash>-<target-filename>` where `<hash>` is the first 8 chars of sha256 of stdin content. Location under graph-dir (not repo root) because sdd already operates within graph-dir.
- Output the saved path in the rejection message so the user can re-pipe it.
- No automatic cleanup — agent handles removal manually. Avoids wiping saves from parallel sessions and keeps the feature simple.
- Add \`.sdd-tmp/\` to \`.gitignore\`.
- If the save fails (permissions, etc.), emit a warning and continue with the normal rejection — don't mask the pre-flight failure.

**\`--dry-run\` flag:**
- New bool flag. Runs graph validation and pre-flight, prints result, exits without writing files or committing.
- Exit code 0 on pass, non-zero on fail — no extra "DRY RUN" framing; exit code is the UX.
- Allowed together with \`--skip-preflight\`: structural validation still runs and is useful on its own.
- Saves stdin to \`.sdd-tmp/\` on every dry-run (pass or fail) using the same scheme. Dry-run never creates an entry, so preserving the heredoc for the follow-up real run is the whole point.

Hash scheme: first 8 chars of sha256 of stdin content. Stable and unique enough for single-user iteration.
