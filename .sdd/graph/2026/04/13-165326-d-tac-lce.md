Mechanical migration of remaining sdd commands and queries to CQRS, continuing from the sdd new reference implementation. Separated from d-tac-lwm into its own plan so a new session can pick up with a focused scope and fresh context budget.

**Reference implementation already shipped** (see handlers/handler_new_entry.go, finders/preflight.go, command/new_entry.go, query/preflight.go, handlers/handler_new_entry_test.go):
- Thin CLI shell in cmd/sdd/main.go newCmd() parses flags → builds NewEntryCmd → constructs Handler with injected deps → calls Handler.NewEntry(ctx, cmd).
- Handler holds deps (graph dir, Preflighter interface, Committer interface, LoadGraph func, stderr, clock).
- Finder owns the PreflightRunner dep and internal orchestration (templates, helpers as lowercase package-internal).
- Results flow back via OnNewEntry callback (id string only — richer data via finder queries per the d-cpt-l3s rule).
- Handler tests use \`package handlers_test\` and exercise only the public surface.
- Three regression tests in handlers/handler_new_entry_test.go honor d-prc-31v (dry-run + validation failure saves stdin; dry-run pass saves once; real-run rejection saves stdin).

**Remaining commands** (writes — handlers):
- \`wip start\` → \`command.StartWIPCmd\` + \`Handler.StartWIP\` (creates marker, optionally creates + checks out git branch).
- \`wip done\` → \`command.FinishWIPCmd\` + \`Handler.FinishWIP\` (removes marker, optionally deletes branch).

**Remaining queries** (reads — finders):
- \`sdd lint\` → \`query.LintQuery\` + \`Finder.Lint\`.
- \`sdd status\` → \`query.StatusQuery\` + \`Finder.Status\`.
- \`sdd show\` → \`query.ShowQuery\` + \`Finder.Show\`.
- \`sdd list\` → \`query.ListQuery\` + \`Finder.List\`.
- \`sdd wip list\` → \`query.WIPListQuery\` + \`Finder.WIPList\`.

**Open design questions to dialogue at session start**:

1. **Presentation layer**: render.go and inline status formatting in main.go — stay in cmd/sdd/ (CLI owns presentation) or move to a presenters/ package? Lean: CLI owns presentation; finders return structured data.
2. **LoadGraph as a finder method**: currently injected into Handler as LoadGraphFunc. Should it become \`Finder.LoadGraph\` so handlers take a single \`*Finder\` dep and get reads + preflight from it?
3. **LoadWIPMarkers**: same question as #2.
4. **Fate of sdd/ package**: once LoadGraph and LoadWIPMarkers move to finders (if chosen), sdd/ becomes near-empty (just render.go, maybe). Delete the package entirely or keep it as an IO compat layer?

**Suggested approach**:
- Start with \`sdd lint\` as the first query migration (smallest scope, easy review).
- Then mechanical migration of the rest — one command/query per commit.
- Each migration: new query/command struct, new handler/finder method, thin CLI shell, test moved or added in the correct package. Handler method tests go in \`package handlers_test\`.
- After all commands/queries migrate, strip any dead code from cmd/sdd/main.go.
- Once complete, a closing action closes both this plan and d-tac-lwm. d-tac-lwm's step 4 (the dry-run regression test) is already satisfied by the three tests in handlers/handler_new_entry_test.go.
