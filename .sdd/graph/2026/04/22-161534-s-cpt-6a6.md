How should SDD handle alternative participant forms when capturing from external material (meeting transcripts, email threads, agent-generated summaries)? The pre-flight rubric shipped in d-tac-6z1 correctly blocks variants at capture time, but real-world sources routinely reference participants under alternative forms — `Christopher Hlubek` in a corporate transcript, `Kate` in Slack, `Mr. Hlubek` in a formal doc.

Two sub-cases, possibly with different answers: (a) the **local participant** has an authoritative canonical via `sdd status` / `.sdd/config.local.yaml`, so agent-side disambiguation is tractable once the agent knows to do it, and (b) **non-local participants** have no equivalent ground truth — the established set from the graph can itself contain drift, leaving the agent guessing whether `Kate` is a shorthand for established `Katherine` or a distinct new voice.

Candidate directions:

1. SKILL.md guidance telling the agent to map external-material forms to the canonical/established spelling at capture time — cheap for local, heuristic for non-local.
2. Extend `config.local.yaml` with an explicit known-collaborators map declaring canonical + alias spellings per person — explicit but adds maintenance.
3. A reconciliation command that clusters variants across the graph and proposes canonical spellings retroactively — catches accumulated drift without per-capture work.
4. A capture-time UX prompt that surfaces candidate mappings to the user for confirmation when ambiguous — trades latency for correctness.

Folding local + non-local into one question because the answer will likely address them jointly via shared infrastructure. No clear answer yet; worth dialoguing when this becomes a felt friction in daily capture.
