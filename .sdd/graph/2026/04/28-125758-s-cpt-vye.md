SDD currently bundles a single agent's skill set (`internal/bundledskills/claude/`) and `sdd init` installs that target for the running contributor. As adoption broadens to teams where contributors use different agents (Claude, Codex, others) on the same project, the data model for agent installation needs to evolve: which agents the project supports may be a project-wide concern (committed in `.sdd/config.yaml`), while which agent each contributor actually installs locally is per-contributor. The shape is open — possibilities include a `supported_agents` list that constrains contributor choices, a per-contributor `target` in `.sdd/config.local.yaml`, or both. Two prerequisites block any concrete design: at least one second-agent skill bundle must exist (today only Claude ships), and cross-agent skill compatibility must be understood (whether the same source serves multiple targets, or each agent needs an authored variant). Deferred from the `sdd init` readiness-check plan (d-tac-kud) since it expands scope onto a single-agent codebase with no second target ready.
