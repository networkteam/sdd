WIP markers for active work tracking. A wip/ directory holds committed marker files indicating in-flight work on graph entries. Markers are mutable working state, not graph entries — they are updated in place and removed when work completes.

Marker file (wip/20260409-103000-christopher.md):

```yaml
---
entry: 20260408-171831-d-cpt-axa
participant: Christopher
exclusive: true
started: 2026-04-09T10:30:00
---

Prototyping widget layout for dashboard. Starting with grid
approach to see if it handles the metric count.
```

Naming: {timestamp}-{participant}.md. The marker ID is its filename, used to reference it in commands.

The exclusive flag controls whether parallel work on the same entry is discouraged (implementing, exploring) or welcome (evaluating, researching). The kind of work is free text in the description — no enum. The playbook guides when to set exclusive.

CLI:
- sdd wip start <entry-id> [--exclusive] [--participant name] <description> — creates marker, auto-commits, warns if exclusive marker exists for that entry
- sdd wip done <marker-id> — removes marker, auto-commits
- sdd wip list — shows all active markers

Integration points:
- Catch-up surfaces markers grouped by exclusive/non-exclusive
- Groom flags stale markers (old, no recent activity)
- SDD skill playbook: implement move creates exclusive marker before starting, removes it when done
- sdd new a --closes <entry> could auto-remove your marker for that entry (implementation convenience, not a design requirement)
