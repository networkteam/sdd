All future framework Go implementation plans must use CQRS as defined in d-cpt-l3s — both as code structure and as a **planning discipline**. The plan itself must decompose new functionality across the layers: what's a CLI action (parsing/formatting only), what's a command (write intent + callbacks), what's a query (read intent), which handler/finder methods process them, what model types are needed. This forces structural choices to be dialogued at plan time, not improvised during implementation. Once the initial migration completes, the surrounding code is the reference and deviations stand out at review. Pre-flight should treat plans that omit the CQRS decomposition — or that conflate layers (side effects outside handlers, handlers being used to query data, rich model data returned from handler callbacks, IO in model) — as structural gaps. This applies d-cpt-vt1's structural-separation principle: pre-flight (as an independent validator) is the gate that enforces architectural compliance, with the CQRS contract giving it concrete criteria to evaluate plans against.
