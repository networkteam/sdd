Voice dictation exposed that SDD conflates two distinct addressing concerns into a single short ID: **internal references** (stable, globally unique, used in frontmatter `refs`/`closes`/`supersedes` and file paths — must survive the life of the graph) and **user-exposed identifiers** (what humans speak, type, or see in rendered output — only need to be unique within a session's working set). These have different requirements, and the current format `{type}-{layer}-{suffix}` is keyboard-optimized for both roles simultaneously.

Voice breaks this conflation across three compounding failure modes: transcription drift ("CPT", "w one m", "dash"), type-prefix drift (speaker remembers content but mislabels d/s/a), and random suffixes that carry no meaning. Live test: dictating "d-cpt-w1m" to reference the bootstrap playbook insight returned "not found" — the actual ID is `s-cpt-w1m` (signal, not decision). Session-opening catch-up numbering already demonstrates the session-scoped user-facing pattern, but it is limited to the catch-up view; mid-session work (grooming, capture, planning, exploration) pulls in entries from outside the current numbered list, leaving no clean handle when title and number both fail.

The voice angle also surfaces a structural question about what the user-facing layer should encode: the current short ID carries type and layer but omits **kind**, which is often more informative at a glance — `plan` vs `directive` vs `contract` vs `aspiration` tells a reader more about an entry's role than `tactical` vs `conceptual` does.

The gap is to separate internal ID stability from user-exposed addressing and rethink what each should communicate. No decision yet; capture as the open gap to revisit when voice becomes a concrete adoption path or when an ID-format rethink is warranted.
