Plan to execute the two-type (signal + decision) redesign with the aspiration kind included, migrate the existing graph to the new type system, and remove all action-type scaffolding from code and data. Operationalizes d-cpt-omm (two-type directive) and d-cpt-xqt (scope expansion to include aspiration) with refinements from session dialogue: activity remains a distinct decision kind alongside directive/plan (WHAT-vs-THAT distinction validated against story and graph evidence); a universal retirement rule covers all entries (supersede by same-kind, or close-by-directive with rationale); short-loop closure (signal-closes-signal) is calibrated by layer and narrative scope; pre-flight validates presence of dialogue-captured reasoning, not its content.

Migration is phased: (1) build two-type support alongside legacy action handling, including a new `sdd rewrite` plumbing command and `--missing-kind` filter; (2) migrate the 98 `a-*` entries to `s-*` + `kind: done`, assign kinds to the 115 kindless signals (default gap), judgment-pass on activity-shaped directives; (3) remove all action-type scaffolding. A fresh `kind: contract` decision stating the two-type system is captured before migration commits, as the target state the rewrite batch migrates toward. Implementation respects the CQRS planning contract (`d-cpt-ah1`): write intents in `internal/command/`, reads in `internal/query/` + `internal/finders/`, side effects confined to `internal/handlers/`, pure domain logic in `internal/model/` — see the plan for the full mapping.

See [plan](./20-113731-d-cpt-e7i/plan.md) for the full spec: kind vocabulary with the question each answers, lifecycle and retirement table, distinguishing tests, short-loop calibration matrix, pre-flight principle, `sdd rewrite` semantics, judgment-pass protocol, and out-of-scope items.

## Acceptance criteria

### Architecture
- [ ] Implementation respects CQRS decomposition per `d-cpt-ah1`: write intents in `internal/command/`, reads in `internal/query/` + `internal/finders/`, side effects confined to `internal/handlers/`, pure domain logic in `internal/model/`, pre-flight templates in `internal/llm/`

### Model & type system
- [ ] Only `type: signal` and `type: decision` are accepted on new entries; `type: action` is rejected after migration
- [ ] Signals carry a `kind`; decisions carry a `kind`; both have defaults when omitted (signal: gap; decision: directive)
- [ ] Signal kinds are validated to one of: gap, fact, question, insight, done
- [ ] Decision kinds are validated to one of: directive, activity, plan, contract, aspiration
- [ ] `done` signals are required to carry `closes` or `refs` on at least one entry (typically a decision; short-loop allows a signal — semantic validation is pre-flight's job, not structural)

### CLI
- [ ] `sdd new s --kind <k>` and `sdd new d --kind <k>` cover all kinds; omitted `--kind` applies the defaults
- [ ] `sdd new a` is removed with a clear migration error pointing to the new form
- [ ] `sdd list --missing-kind` filters entries without `kind`
- [ ] `sdd rewrite <id> --type <t> --kind <k>` atomically renames the file, rewrites frontmatter, updates all inbound references, and commits the change as one unit; supports `--dry-run` and `--no-commit`
- [ ] `sdd status` surfaces the expanded vocabulary — aspirations distinguishable from contracts, activities distinguishable from directives

### Pre-flight & validation
- [ ] Structural validation (kind field presence, plan AC section presence, done-signal `closes`/`refs` is non-empty, refs resolve) runs as typed errors in the model / CLI layer before LLM pre-flight is invoked
- [ ] LLM pre-flight templates stay organized per check transaction, not per kind
- [ ] `closing_action.tmpl` renamed to `closing_done.tmpl` with action references removed
- [ ] `action_closes_signals.tmpl` renamed to `short_loop.tmpl`; retargeted for done-kind signal closing a signal; applies the layer × scope-shape calibration matrix with gap-coverage and smuggled-decision checks
- [ ] New `dissolution.tmpl` applied when fact or insight closes a question; checks dialogue-captured context presence (not content-correctness)
- [ ] New shared partial `unrelated_refs.tmpl` invoked by all capture templates to flag topically disconnected refs (per `s-prc-uy2`)
- [ ] New shared partial `unusual_close.tmpl` invoked by all close-carrying templates; surfaces atypical close patterns with severity gated by rationale presence — `high` when narrative lacks rationale, `low` when rationale is clear, no finding on standard patterns
- [ ] `closing_decision.tmpl` checks for retirement rationale when the target is stable-kind (fact, insight, contract, aspiration)
- [ ] `decision_refs.tmpl` surfaces a directive-reads-aspiration-shaped finding (medium/low) when `kind: directive` at stg/cpt layer lacks a plausible completion criterion
- [ ] New `aspiration_capture.tmpl` is its own template applied when `kind: aspiration` is captured; gradient findings at low or medium severity only (never high)
- [ ] Every new or modified calibration band names a concrete observable pattern with at least one example; every template includes an explicit "no finding" enumeration (matches discipline in existing `signal_capture.tmpl`, `closing_action.tmpl`, etc.)
- [ ] `sdd rewrite` bypasses pre-flight

### Migration
- [ ] A `kind: contract` decision stating the two-type system is captured before any migration rewrites commit
- [ ] All 98 existing `a-*` entries are rewritten to `s-*` with `kind: done`; every inbound `refs` / `closes` / `supersedes` reference is updated
- [ ] All 115 existing kindless signals carry a `kind` (gap default; fact / question / insight where narrative is unambiguous)
- [ ] Activity-shaped `kind: directive` entries from before the plan are reviewed in a judgment pass and re-kinded to `kind: activity` where appropriate
- [ ] `sdd lint` reports clean after the migration batch completes
- [ ] `sdd summarize --all` runs as the final migration step

### Teardown
- [ ] No `TypeAction` constant (or equivalent) remains in the codebase; the model recognizes only signal and decision types
- [ ] No entry in the graph carries `type: action`
- [ ] CLI help, README, and skill documentation reflect the two-type / five-kind vocabulary
- [ ] Skill instructions describe retirement primitives, short-loop calibration, and the directive / activity / plan distinguishing tests

### Out of scope (explicit)
- Retro-labeling strategic directives as aspirations — handled as follow-up actions after the aspiration kind lands (e.g., s-cpt-wiv #4 closing s-stg-gtu)
- Additional kind proposals beyond the five settled here — future additions go through the normal dialogue-before-capture loop
