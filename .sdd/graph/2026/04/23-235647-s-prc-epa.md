`sdd init`'s post-install commit can unexpectedly sweep up unrelated staged code changes. Running `./bin/sdd init --scope project` during the `d-cpt-d34` implementation committed 25+ staged files across `internal/` and `cmd/` under the message `sdd: refresh installed skills and metadata` — the init handler's commit step doesn't limit paths to what it wrote. The resulting commit (f3bc338) is content-correct but message-misleading and muddles review / bisect.

Two fix directions worth dialoguing:

- **Scope the commit to only the skill files init touched** — the handler knows which paths it wrote, so it can pass them explicitly rather than relying on whatever else is staged.
- **Skip the auto-commit when the working tree has unstaged changes outside `.claude/skills/`** — defensive: surface a warning and let the user opt in explicitly via a flag, so ambient staged work isn't captured under a misleading message.

Observed in the `d-cpt-d34` implementation session — an easy workaround is "don't stage other work before running init," but that's discipline, not a structural guarantee.
