Done signals treated as terminal is an operational simplification, not a framework truth. Mechanical validation (unit tests, lint, CI) is a precondition of the done signal — the "I can claim this is done" baseline that happens *before* capture. The loop's broader aspiration is a further step: implementation → evaluation (inner — code review, structural observation; outer — E2E, UX, production behavior) → new signals (gaps, insights, facts). A done signal captures that implementation landed and mechanical checks passed, not that evaluation happened.

**Idea for future refinement:** "done done" confidence could emerge from the downstream pattern rather than a new state. A done signal with positive downstream refs — inner-evaluation insights (code review findings, structural observations), outer-evaluation facts (E2E, UX, production behavior), or done signals on explicit evaluation activities — accumulates proof. The graph already supports this via `refs`; it's a reading convention, not a schema change. A done signal with no evaluation downstream is "committed but unvalidated"; one with corroborating downstream is "done done."
