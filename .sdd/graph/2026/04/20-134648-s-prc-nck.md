Agent proposed pre-flight template design (per-kind templates like `gap.tmpl`, `fact.tmpl`, etc.) during the d-cpt-e7i planning session without first reading existing pre-flight templates in `internal/llm/preflight_templates/`. The proposal contradicted the established per-transaction organization and would have re-invited the over-correction lineage (s-prc-2nh, s-prc-q2m) if captured directly. User prompted the agent to check; agent then found the existing pattern and folded the full seven-signal tuning history into the plan's Calibration discipline.

Pattern: agent proposing infrastructure changes (templates, CQRS layers, routing) without grounding in existing code is another instance of the instruction-drift family (s-stg-3vr, s-prc-w5r, s-prc-cgk). Notable for happening INSIDE a planning session where grounding is especially load-bearing — the plan codifies, and un-grounded propositions compound into architectural debt.

**Idea to explore:** could the `/sdd` planning playbook tackle this structurally? Two angles surfaced in dialogue: (a) an explicit **grounding rule** — every proposition in a plan (kind sets, template organization, routing logic, CQRS decomposition, CLI surface) should be traceable to an existing decision, code reference, or confirmed dialogue turn, not freshly invented; (b) a **pre-capture validation pass** — before `sdd new d --kind plan`, the agent walks each significant proposition with the user and names its grounding source, so un-grounded items get caught before the plan codifies. Both are ideas; phrasing, placement, and scope belong to a future decision that addresses this signal.
