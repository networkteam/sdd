Evaluating the participant identity enforcement shipped earlier today (s-tac-cbu, commit d9165c5) against its first real post-landing capture surfaced a coverage gap: two done signals (s-tac-x9m, s-tac-d3u) passed pre-flight with "Christopher Hlubek" instead of the canonical "Christopher" from `.sdd/config.local.yaml`. The feature designed to catch this exact class of drift missed it on first use.

Two compounding failure modes. **Validator-side**: `participantDriftFindings` (`internal/finders/preflight.go:45`) does exact-string match against `graph.AllParticipants()`, which includes same-session drift — today's morning-session aspirations (d-stg-beb through d-stg-2wb) had already introduced "Christopher Hlubek" pre-landing, so the variant registered as legitimate. Config's canonical "Christopher" is ground truth for the local participant but never reaches the check. **Agent-side**: when composing `--participants` with Claude as an additional voice, the agent pattern-matched on today's `sdd status` output rather than config or the longer-established spelling, violating explicit skill guidance ("prefer established spellings") — another instance of the instruction-drift cluster (s-stg-3vr, s-prc-w5r, s-prc-cgk, s-prc-nck) where text-layer rules alone fail to control agent behavior.

Structural fix direction dialogued: config canonical becomes an authoritative input to the validator, and the local canonical is surfaced in agent-consumed context (status output header) so ground truth rides along automatically — per d-cpt-vt1's structural-separation principle. The drifted entries will be mechanically corrected in place per d-cpt-e1i (meaning preserved, not superseded).
