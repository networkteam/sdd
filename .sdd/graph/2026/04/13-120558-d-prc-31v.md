Every bug fix requires a regression test, regardless of how the bug was reported (pre-flight, manual review, production, post-merge evaluation, etc.). The test must fail before the fix and pass after, and must land in the same change as the fix. Test-first is advised when practical (write the failing test, implement the fix, see it go green) — it provides immediate proof the test exercises the right gap. When test-first is impractical (e.g. the test infrastructure itself needs to be built first), the test must still accompany the fix; "I'll add it later" doesn't count. Builds on d-cpt-vt1: structural separation (independent reviewers, pre-flight gates) catches bugs at the moment they happen — the test converts that one-time catch into a permanent barrier so the same regression class can't recur. Corollary: if a fix is genuinely hard to test, treat that as a signal that the code's structure should change, not as a justification for skipping the test.
