Participant names are used inconsistently across the graph. An April tally of the 2026/04 entries shows roughly even split: "Christopher Hlubek" (130), "Christopher" (109), "Claude" (195), and — notably — 30 entries with a leading-space variant " Claude", which points to a CSV-parsing bug where `--participants "Christopher, Claude"` is split without trimming whitespace. Without a source of truth for the user's canonical name, each session infers from context (gitStatus, prior mentions) and can land on a different form, which undermines participant-based queries and narrative reading of the graph.

Four concrete fixes surfaced in dialogue:

1. **Trim whitespace** in the `--participants` CSV parser so " Claude" no longer leaks in as a separate identity. Mechanical bug fix, orthogonal to the naming problem.
2. **/sdd-catchup lists active participants** drawn from recent entries, so the model sees established usage before constructing `--participants` for new entries.
3. **Local `whoami` config** (via `sdd meta` or equivalent) giving the CLI/skill a canonical participant name to pass via `--participants` by default. The config is local deliberately — different checkouts on the same machine or different team members sharing a graph should set their own identity without committing it.
4. **Pre-flight validates participants against a known list** derived from prior entries. Calibration: flag as medium when a new unknown participant enters without prior dialogue introducing them (legitimate new collaborators should be discussed before first appearance); flag as high when a participant's spelling looks like a near-miss of an existing one (e.g. "Chirstopher" vs "Christopher" or "Christopher" when "Christopher Hlubek" is the established form) — suspicious typo, not a new person.
