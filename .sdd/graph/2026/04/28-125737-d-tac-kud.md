Make `sdd init` the canonical readiness check for SDD-instrumented repos by persisting the install scope in `.sdd/config.yaml`, presenting an interactive selector when the choice is undecided, and having the command idempotently detect and prompt for missing setup pieces — local participant, skills at the recorded scope — on any run. The plan also extends the existing `minimum_version` write gate to cover `sdd init` itself and adds an explicit `--bump` flag for raising the floor on demand. After this plan lands, the instruction "after cloning, run `sdd init`" is a complete answer for new contributors: no manual file editing, no guessing where skills should live, no half-installed state when binary versions don't line up.

The plan resolves both fresh-clone setup (s-tac-z8o) and scope persistence (s-tac-eo7), which aren't independently implementable: detecting "skills missing on a fresh clone" requires knowing where they were supposed to be installed, and persisting scope only pays off when `sdd init` consumes it. It also closes a previously-unguarded edge in the version-skew infrastructure shipped under s-tac-3al — the dispatcher write gate covers mutation commands but not `sdd init` itself, leaving a path where an older binary's bundled skills could overwrite a newer install on a clone. Scope contradictions block rather than rewrite: an explicit `--scope` flag disagreeing with the recorded value errors and points the contributor at `.sdd/config.yaml` for direct edit. Multi-agent (Claude vs Codex etc.) support is deliberately deferred — see the attachment for rationale and the companion conceptual gap signal.

See [design notes](./28-125737-d-tac-kud/plan.md) for alternatives considered, out-of-scope items with rationale, and the relationship to the separate `sdd init` post-install commit hygiene gap (s-prc-epa).

## Acceptance criteria

- [ ] `.sdd/config.yaml` carries a `skill_scope: project|global` field; `sdd init` writes it the first time scope is chosen and preserves it on subsequent runs.
- [ ] An explicit `--scope` flag that contradicts the recorded `skill_scope` value blocks with a clear error naming the conflict and pointing at `.sdd/config.yaml` for manual edit.
- [ ] When no `skill_scope` is recorded, `sdd init` presents a Bubble Tea selector (project / global, default highlighted) for the contributor to choose. Non-interactive contexts (no tty on stdin) error with `--scope` flag guidance instead.
- [ ] `sdd init` runs idempotently on any starting state (fresh clone, partial setup, fully set up): detects a missing local participant and prompts; detects missing skills at the recorded scope and re-installs.
- [ ] `sdd init` in a non-interactive context errors with a single message naming each missing piece and the exact flags needed to fix it non-interactively (`--language`, `--scope`, `--participant`).
- [ ] `sdd init` is wrapped by the existing `withWriteGate` (or an equivalent compatibility check) so a binary below the graph's `minimum_version` is refused with the same upgrade-guidance error mutation commands already produce, before any skill files are written.
- [ ] `sdd init --bump` writes `minimum_version` in `.sdd/meta.json` to the running binary's version when the binary is higher than the recorded value (or when no value is recorded); is a no-op when values are equal; errors with "cannot bump from a dev build, use a released sdd binary" on dev builds.
- [ ] Other `sdd` commands emit a single one-line warning to stderr when local participant is missing, naming `sdd init` as the fix. The warning is suppressed for `sdd init` itself and for structured / `--quiet` output modes.
- [ ] README Quickstart names "run `sdd init`" as the explicit first step after cloning an SDD-instrumented repo, ahead of any other workflow.
