Voice exploration surfaced a candidate user-facing addressing pattern: **semantic topic label + kind suffix**, replacing the current keyboard-optimized `{type}-{layer}-{suffix}` short ID as the spoken/written handle. The insight rests on two observations: (1) *kind subsumes type* — the 5+5 signal and decision kinds are disjoint, so naming the kind (aspiration, contract, plan, directive, activity, gap, fact, question, insight, done) carries the type implicitly while being more informative about the entry's role; (2) *layer is rarely load-bearing for identification* — topic usually implies it. So `topic + kind` strictly dominates `type + layer + random-suffix` as a user-facing handle.

A live voice test resolved three references through the pattern: "the actor roll-out plan" → `d-cpt-thg` (despite a dropped word, resolved via `plan` kind-filter + topical match on "actor"), "the first-class actors directive" → `d-cpt-pc0`, "the focus layer gap" → `s-cpt-cca`. The resolution model: kind acts as a type-filter narrowing candidates, topic does fuzzy match, full ID stays canonical. Labels can collide; ambiguous references disambiguate at lookup through dialogue.

**Architectural boundary**: the CLI continues to accept only full IDs and short IDs; semantic label resolution is strictly an agent-layer responsibility. Labels appear in rendered CLI output (so humans can see them and know what to reference in voice or dialogue) but are not accepted as CLI input — the agent resolves labels to IDs before invoking commands. This preserves CLI determinism and scriptability, avoids shell-quoting for space-separated labels, and locates fuzzy matching (kind filtering, topical match, collision disambiguation) in the layer that naturally handles ambiguity through dialogue.

Open design questions deferred: whether to always render the kind suffix or only on collision; whether labels are generated at capture (LLM-produced, stored in frontmatter) or derived (deterministic from ID); and how the pattern interacts with concurrent capture — where two participants could independently choose the same topic label, creating a new class of merge inconsistency the current ID format precludes by construction, which would need post-merge lint tooling (extending `sdd lint`) with resolution strategies like first-by-date-wins-with-numeric-suffix or regenerate-with-context.
