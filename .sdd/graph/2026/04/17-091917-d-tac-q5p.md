Resolve participant-name drift through three coordinated pieces that close out the remaining scope of s-prc-omw after a-tac-1e6 landed the trim fix and catch-up surfacing.

Root cause: implicit name handling. Agents infer participants from session context (e.g. `git user.name` = "Christopher Hlubek") and the graph already contains the canonical form ("Christopher"). Without explicit storage + correlation, each new entry re-rolls the dice.

Scope:
1. Local config — `.sdd/config.local.yaml` stores a `participant` field. `.local.yaml` is the convention (gitignored, extensible for future local settings). `sdd init` prompts interactively with `git user.name` pre-filled as the default; accept or override. `sdd new` / `sdd wip start` fall back to the local config when the flag is omitted; error cleanly if no config is set.
2. Skill correlation rule — `/sdd` playbook adds: before capture, correlate proposed participants against existing names visible in `sdd status` (per-line since a-tac-1e6) and prefer established spellings. Introducing a new voice requires confirming with the user rather than guessing.
3. Pre-flight drift finding — new medium-severity category that fires when an entry's participants include a name not matching any name already in the graph. The finding lists established names so the LLM / author can decide typo vs new voice. Safety net when agent correlation fails.

Design choices:
- Local config over global (`~/.sddrc`) — participants are project-scoped; same person may be "Chris" in one graph, "Christopher" in another.
- `config.local.yaml` naming over `whoami` — clearly local (gitignore convention), extensible for future non-participant settings.
- `sdd init` is the canonical "catch me up" command — re-running applies any housekeeping (gitignore entries, schema additions, directory creation) idempotently, so upgrades require no separate migration command.
- No silent fallback to `git user.name` at capture time — the whole point of the config is to fix the drift git.name introduces. Prompt at `sdd init` with git.name as default; error cleanly at capture if config is missing.
- Explicit `--participants` / `--participant` flags are pure override — the flag is used exactly as given; local config is NOT implicitly merged. Matches `--refs` semantics; keeps "capture for someone else's session" unambiguous.
- Pre-flight over CLI parse time for drift — it's semantic judgment (typo vs new voice), not presence. Consistent with s-prc-hpa's distinction (deterministic checks → CLI; judgment → pre-flight).
- Medium severity, not high — new voices are legitimate (meetings, new collaborators); gatekeeping would be user-hostile.

## Acceptance criteria

- [ ] `.sdd/config.local.yaml` supports a `participant` field
- [ ] `sdd init` interactive prompt for participant pre-fills `git user.name` as the default and writes it to `.sdd/config.local.yaml`
- [ ] `sdd init --participant <name>` writes the config non-interactively (for scripts / CI)
- [ ] `sdd init` is idempotent: re-running applies all housekeeping steps (gitignore entries for `.sdd/config.local.yaml` and `.sdd/tmp/`, directory creation, any newly added config schema fields) without duplication or data loss; existing values pre-fill prompts as defaults; unknown fields in the file are preserved
- [ ] `sdd new` uses local-config `participant` for `--participants` when the flag is omitted; when the flag is present it is used exactly as given (pure override — local config is NOT implicitly merged); missing flag + missing config exits non-zero with "run `sdd init`"
- [ ] `sdd wip start` uses local-config `participant` for `--participant` under the same rules as `sdd new`
- [ ] `/sdd` skill text adds a rule: before playback, correlate proposed participants against names visible in `sdd status` and prefer established spellings; ask the user before introducing a new voice
- [ ] A new pre-flight finding category "participant-drift" fires at medium severity when an entry's participants include a name not matching any participant already in the graph
- [ ] The drift finding lists established names from the graph so author / LLM can judge typo vs new voice
- [ ] Multi-participant entries (e.g. "Christopher,Claude") are validated per-name, not as a single concatenated string
