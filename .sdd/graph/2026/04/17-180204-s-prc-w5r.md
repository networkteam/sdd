Capture flow breaks when agents reflexively run `sdd new --dry-run` before the real capture, even for content already aligned through dialogue. Observed during the capture of d-tac-lg1 (release pipeline plan): the agent invented a "dry-run first" rule not present in the skill, then with stdin attach (`--attach -:plan.md`) the `--dry-run` saved stdin to `.sdd/tmp/` and printed a retry hint that round-trips stdin (`cat <tmp> | sdd new ... --attach -:<target>`), which re-enters the same code path if followed literally. The agent then misread silent preflight output as a short-circuit bug, requiring several user-correction rounds before reading `internal/handlers/handler_new_entry.go` and confirming preflight actually ran — zero findings just produced no output.

Two underlying issues:

1. **Unwarranted dry-run reflex.** SKILL.md already prescribes capture-once-with-preflight (lines 41-45: high blocks, medium surfaces non-blocking, low is informational). Dry-run adds value only when content is genuinely uncertain; for well-dialogued content it's ceremony that breaks flow. The agent manufactured the ceremony and attributed it to the playbook — a fresh instance of the instruction-drift pattern in s-stg-3vr.

2. **Retry hint cosmetics.** The hint at `internal/handlers/handler_new_entry.go:50` suggests `cat <tmp> | sdd new ... --attach -:<target>` — a stdin round-trip that re-saves to the same hash path. `--attach <tmp-path>:<target>` would be the cleaner default suggestion, avoiding the re-save cycle and making the workflow obvious.

Flow-break cost this session: two wasted `sdd new` invocations, a confident-but-wrong bug claim, and multiple correction round-trips before investigation. Calibration signal for both agent behavior (don't manufacture ceremony beyond what the skill specifies) and CLI UX (retry hint should point at the simpler path).
