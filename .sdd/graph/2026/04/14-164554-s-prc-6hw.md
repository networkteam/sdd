Pre-flight validation disrupts flow more than it protects quality. In this session alone, pre-flight rejected entries 3-4 times before acceptance — each rejection costs 60-120 seconds of wall-clock time plus context-switching overhead for rewording. The rejections fall into two categories: genuinely useful catches (confidence level mismatch, weak first sentences) and flow-killing overreach (demanding deviation markers for user-approved scope splits, arguing with scope decisions the user already made, timing out on large closing actions). The cost-benefit is wrong: a tool that blocks the user 3 times per entry to catch 1 real issue is a net negative on velocity. Compounding factor: haiku's 120-second timeout is too tight for closing actions with many refs — the prompt grows with each referenced entry, and complex entries regularly hit the timeout. The pre-flight contract (d-prc-31v) and the over-application signal (s-prc-okf) both feed into this: the system enforces rules too broadly and too slowly. Possible directions: make pre-flight advisory (warn, don't block), reduce prompt size by using summaries for referenced entries, add a fast-path for entries where the proposing agent already dialogued the content with the user.
