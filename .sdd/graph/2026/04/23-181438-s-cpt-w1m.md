The most load-bearing scaffold for SDD adoption under the focus directive (`d-stg-u2i`) is a generic bootstrap playbook covering project onboarding — first participant captures, first strategic and conceptual decisions, orientation dialogue, and the sequencing that turns a fresh graph into a working one — rather than any single exploration path (multilingual, actors/roles, background sync) individually. All three named explorations presuppose a "get a project going" flow that currently exists only as mechanical `sdd init`; the dialogue-driven content is implicit in the `/sdd` skill but not formalized as an adoption artifact.

The need is immediate, not speculative: this graph itself already has 353 entries with `participants` fields but no actor signals — running the playbook on this repo is the first concrete test case, not a future scenario. Shaping the playbook now serves both the current bootstrap need (capture the two existing actors properly) and future adoption (reusable scaffold for any project).

Recognized while shaping the actors/roles sub-plan: the "who captures actors when" question kept surfacing the broader "how does a project start" question. Framing it as a bootstrap playbook recenters adoption work on the generic scaffold that any adoption scenario needs, rather than per-scenario mechanisms (actor migration, multilingual setup, etc.) that would each want their own playbook. First-actor capture, first-decision capture, first-aspiration dialogue — all are instances of the same "bootstrap" pattern.

Suggests sequencing: hold the actors/roles implementation plan while shaping the bootstrap playbook, since the playbook will likely absorb the first-actor-capture concern rather than living alongside it. Grounded in the dialogue-first aspiration (`d-stg-beb`): mechanical `sdd init` surfaces the starting need, the bootstrap playbook is where the dialogue-driven content lives.
