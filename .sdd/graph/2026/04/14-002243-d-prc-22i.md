Move action-durability enforcement from a graph contract into pre-flight's `closing_action.tmpl`, superseding `d-prc-zch`.

The superseded contract required actions to be captured after the work they describe is committed to Git. Two blind spots surfaced: (1) knowledge-capture actions where `--attach` brings in research or synthesis artifacts that become durable via the entry's own commit — observed during `a-cpt-xbv`, which needed `--skip-preflight`; (2) split knowledge capture where an action references artifacts already durable in an upstream entry. Meta-problem: the rule is mechanical, not judgment-requiring — pre-flight checks it on every closing action, humans don't reason about it. Real contracts (`d-cpt-vt1`, `d-prc-31v`) ask for judgment; this one asks for a sequence. It belongs in the validator's prompt.

Durability can arrive via three paths, any of which pre-flight accepts as presence evidence: prior commit referenced by the action (work-recording case); attachments on this entry via `--attach` (single-session knowledge capture); attachments on a referenced upstream entry already in the graph (split / multi-session knowledge capture). Enforceability is asymmetric — only code commits are truly verifiable; attachments and upstream refs are presence-checked only; external artifacts remain trust-based and outside scope. The contract will not pretend to enforce what it cannot verify.

This is a directive, not a plan, because superseding a contract with a plan is structurally inconsistent (pre-flight flagged this on first attempt). The directive retires `d-prc-zch` outright — no replacement contract — and moves the check into pre-flight's prompt template. `--skip-preflight` used because retirement-without-replacement is an unhandled scenario in the pre-flight validator (not a flaw in this decision).

See [design](./14-002243-d-prc-22i/design.md) for the three-path taxonomy, the proposed `closing_action.tmpl` wording, and the SKILL.md line 60 replacement.
