SDD's core design — graph format, CLI abstractions, skill structure, and primary distribution channels — should remain agent-agnostic, without any single-agent assumptions baked into the foundations. MVP implementation may legitimately target Claude Code as the sole supported agent; shipping Claude-only features, integrations, or skills does not violate this direction. The distinction is deliberate: the neutrality discipline applies to what's expensive to change later (core graph format, primary distribution path, central CLI abstractions, skill-content authoring format), not to what's cheap to add later (agent-specific features, per-agent integrations, Claude-tuned skill variants). Concretely at MVP: distribution uses agent-neutral channels (Homebrew, curl|bash, GitHub Releases) rather than any agent's marketplace as the primary path; the sdd install command is shaped to support multi-agent target selection even if only Claude is wired up; skill content is structured so per-agent variants can be derived later without rewriting the source. Extending to Cursor, Codex, Gemini CLI, or future agents should be a natural addition on top of the foundations, not a rewrite. Provenance and grounding: this directive is a new strategic commitment that emerged during session dialogue while evaluating s-cpt-4gj, not a response to a pre-existing signal specifically calling for agent-neutrality. It is grounded in (a) the sixth angle of a seven-angle research pass on distribution tooling (see companion research action), which argued that conventional package-manager distribution plus config-layer agent integration beats native-marketplace-primary for CLI tools in 2026, citing CLIs' longer lifespan than individual AI products, emerging convergence at the skill-package level rather than marketplace level (Cursor, Copilot CLI, Gemini CLI all ship skills-like constructs), and marketplace-governance risk from single-vendor coupling; (b) the landscape observation in s-stg-gtu that SDD operates alongside a growing set of AI memory/coordination tools, making portability a positioning concern; and (c) the adoption pathway framing in s-stg-qg0, where neutrality enables broader adoption by not coupling SDD's reach to any one agent's user base. During dialogue the user introduced the MVP-vs-neutrality distinction — 'in our current MVP phase it should not be prohibitive to just build features for Claude' — which reshaped the directive from a strict anti-lock-in rule into its current cheap-vs-expensive-to-change shape: foundations stay neutral, feature surface can be agent-specific. Starting as a directive (not kind: contract) by explicit choice — 'can come later if we want to be more strict here' — reserves promotion to contract for when pre-flight enforcement becomes warranted. Scope boundary: this directive addresses agent-platform neutrality specifically (Claude Code vs Cursor vs Codex vs Gemini CLI) and is orthogonal to separate open concerns about access modality (CLI vs chat UI, raised in s-tac-7kh) and retrieval-tool positioning (raised in s-stg-gtu); neither is resolved by this directive. Relationship to s-cpt-4gj: that signal framed distribution as a choice between three options (Claude Code plugin, embed skills in binary, curl|bash installer). This directive partially answers by ruling out any single-agent-marketplace path as primary, on strategic grounds. The concrete channel choices (specific tools, versioning, schema compat) are made at the conceptual layer as a separate directive, not forward-referenced here. Alternative considered and rejected: Claude-native-first with portability deferred — risks baking assumptions into foundations that become expensive to unwind, which is exactly what the cheap-vs-expensive-to-change distinction is designed to prevent.
