Unify skill installation under idempotent `sdd init` — one command handles first-time project setup, skill extraction, skill refresh after binary upgrades, and graph schema drift detection. Drops the separate `sdd install` subcommand planned in d-cpt-uu1 in favor of `terraform init`-style "run whenever things change" semantics.

**Why:** A two-command design (`init` + `install`) created an onboarding trap — a new user `sdd init`'s a project, opens Claude Code, types `/sdd`, and finds nothing because skills were a second command away. Separation added vocabulary without user benefit; skills and projects share lifecycles in the common case (first-time setup, version refresh).

**In scope:** skills embedded via `go:embed` and extracted by `sdd init` with injected `sdd-version` + `sdd-content-hash` frontmatter stamps; pristine-hash refresh UX (auto when content matches stored hash, prompt when locally modified); `.sdd/meta.json` (committed) with `graph_schema_version` plus `minimum_version` written on initial init and preserved thereafter; write-gate on `sdd new` / `sdd wip` / `sdd summarize` checking both fields; reads unrestricted.

**Deferred:** `sdd migrate` and migration chain (no v2 schema yet); `sdd set-min-version` command (hand-edit sufficient for MVP); `--target <agent>` abstraction (Claude hardcoded until a second agent is wired).

See [plan](./18-151212-d-tac-r5g/plan.md) for design details, CQRS decomposition (per d-cpt-ah1), alternatives considered, open implementation questions, and sequencing.

## Acceptance criteria

- [ ] Fresh-project `sdd init` creates `.sdd/config.yaml`, graph dir, and `.sdd/meta.json`
- [ ] `sdd init` writing a new `.sdd/meta.json` sets both `graph_schema_version: 1` and `minimum_version` to the binary's own semver version
- [ ] `sdd init` on an existing `.sdd/meta.json` with `minimum_version` set preserves it unchanged; never overwrites
- [ ] Dev builds (non-semver version strings) skip writing `minimum_version` on initial init; `graph_schema_version` is still written
- [ ] Skills embedded via `go:embed`; `sdd init` extracts them to the target directory with per-skill `sdd-version` frontmatter matching the binary version
- [ ] Each installed skill file carries `sdd-content-hash` — hash of the file with `sdd-version` + `sdd-content-hash` themselves stripped and frontmatter canonicalized
- [ ] Default target is `~/.claude/skills/`; `--scope project` redirects to the repo's `.claude/skills/`
- [ ] Repeat `sdd init` with matching binary version and pristine hashes is a no-op (no prompts, no writes)
- [ ] Repeat `sdd init` after binary upgrade: installed hash matches stored `sdd-content-hash` → refresh silently; mismatch → prompt before overwriting
- [ ] Write commands (`sdd new`, `sdd wip`, `sdd summarize`) check `graph_schema_version` and `minimum_version` (if present) on startup; schema mismatch errors point at `sdd init` or future `sdd migrate`; binary below `minimum_version` errors with upgrade guidance
- [ ] Read commands (`sdd status`, `sdd show`, `sdd list`) are not gated by either field
- [ ] Dev-build binaries bypass both write gates (treated as matching anything)
- [ ] Interactive prompts reuse the Bubbles textinput infrastructure from d-tac-s2g
- [ ] README.md, `docs/`, and `.claude/skills/` describe the unified `sdd init` flow with no references to a separate `sdd install` command; d-cpt-uu1's original text remains intact per the immutability contract (d-cpt-e1i)
