Engaging with SDD today starts with an explicit `/sdd` invocation — the participant types the slash command to enter the graph-guided catch-up flow. Once inside, the session runs ambient: dialogue drives the playbook, sub-skills (`sdd-explore`, `sdd-groom`) dispatch transparently when playbook moves call them, and capture flows through natural conversation. The mode-switch is at the entry point, not inside the session.

Against the ambient-capture aspiration (`d-stg-2wb`), which pulls toward SDD emerging from the work rather than being entered, this entry-point gate is a concrete gap. What more-aligned tooling would look like: skills firing from dialogue cues in any ongoing conversation (a user saying "let's capture this" triggers capture without `/sdd` first being active), graph reads surfacing when an existing discussion nears a captured signal, catch-up content appearing on relevant context changes rather than on explicit invocation.

The present pattern works and is needed while dialogue-cue triggers mature — this isn't a bug report. It's a direction for future tactical decisions to drive toward.
